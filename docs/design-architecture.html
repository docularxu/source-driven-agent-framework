通用代码分析/逆向工程架构 - 与具体项目无关 - Guodong × Jarvis - 2026.3.10

0. 与开发框架的区别

🔨 开发框架 (Spec-Driven Dev)

产出：代码
Source of truth：Spec 文档
Reviewer 检查：代码是否符合 spec
任务拆分：按 feature / user story
流程核心：No Doc, No Code

🔬 分析框架 (Source-Driven Analysis)

产出：分析文档
Source of truth：源码本身
Reviewer 检查：分析是否准确、完整
任务拆分：按模块 / 子系统 / 调用链
流程核心：No Evidence, No Claim

核心原则：No Evidence, No Claim
每一条分析结论都必须附带代码引用（文件:行号）。
看不懂的标注"待分析"，不猜测。
分析文档中没有代码引用的结论是危险信号。

2. 分析驱动流程 (Source-Driven Analysis)

2.1 三阶段流程

Phase 1: 侦察与规划 (Reconnaissance)
Step 1 - 目标对齐：Guodong 提出初始分析目标（可能模糊），Jarvis-Arch 通过直接对话反复追问、澄清模糊地带，双方确认分析范围与边界后再开始侦察。必须确认的项包括：分析范围、关注点、已知入口、目标编译配置（defconfig / 关键 CONFIG 选项，决定 #ifdef 分支走向）。
Step 2 - 代码侦察：Jarvis-Arch 对代码库做初步侦察（目录结构、构建系统、入口点识别），产出 PROJECT.md（项目概览 + 模块拆分 + 分析计划）+ SYMBOL_INDEX.md（全局符号表）。
Step 3 - 计划审查：Reviewer 审查分析计划的合理性（模块边界是否清晰、有无遗漏的子系统）。
Step 4 - Sign-off：Guodong 确认后进入 Phase 2。

与开发框架的区别：Phase 1 不是写 spec，而是画地图 - 搞清楚要分析的领地有多大、从哪里入手。目标对齐是第一步，因为分析方向错了后面全白做。

⚠️ 编译配置是分析前提，不是可选项。C 代码库中大量逻辑隐藏在 #ifdef 分支里（如 CONFIG_SPL_BUILD、CONFIG_ARM64）。如果不在 Phase 1 就锁定目标编译配置，Researcher 和 Reviewer 可能在不同的 #ifdef 分支上工作，导致分析结论互相矛盾却都"正确"。Jarvis-Arch 必须在 Step 1 与人类工程师确认 defconfig 和关键 CONFIG 选项，并记录到 PROJECT.md 中，作为后续所有分析和验证的共同基准。
Phase 2: 逐模块深度分析 (Deep Dive)
Jarvis-Arch 派发分析任务 → Researcher 读代码、输出分析文档 → Reviewer 验证准确性。
每个模块走完 Researcher + Reviewer 后，Jarvis-Arch 整合进全局知识图谱。
全部模块完成后进入 Phase 3。

调用链追踪深度限制：Researcher 追踪调用链超过 task.md 指定的层数（默认 5 层）仍未到达边界时，必须暂停并向 Jarvis-Arch 汇报当前位置和困难，由 Jarvis-Arch 决定是继续深追、缩小范围、还是标记 [PENDING] 跳过。禁止无限追踪。
Phase 3: 整合报告 (Final Report)
全部模块分析完成后，Jarvis-Arch 基于所有分析文档和最终版知识图谱，整合最终报告。
Guodong sign-off 后项目结束。

📥 输入：
- 所有 analysis/*.md
- knowledge-map.md（最终版）
- retro.md（全部复盘记录）

📤 输出：
- REPORT.md - 最终整合报告（可发表）

✅ 完成标志：Guodong sign-off

2.2 分析任务模板 (task.md)

# 模块名称
## 分析目标 （这个模块干什么、为什么要分析它）
## 入口点 （从哪个函数/文件开始追、已知的调用者）
## 分析范围 （追到哪里停、哪些分支不用追）
## 关注点 （重点关注什么：数据结构？寄存器配置？开源/闭源边界？）
## 动态分发/隐性跳转 （已知的函数指针、ops 结构体、宏替换、__weak 覆盖。要求 Researcher 遇到静态无法解析的跳转必须在此记录，不得跳过或猜测目标）
## 上下文 （依赖前序模块的哪些结论、全局知识图谱中的相关信息）
## 产出要求 （调用链图、数据结构说明、待分析清单）

2.3 分析文档产出格式

Researcher 每次分析输出一个 markdown 文档，必须包含以下结构：

# 模块名称 - 分析报告

## 概述 （一句话说明这个模块的功能）

## 调用链
（从入口函数逐层展开，每个函数附 文件:行号）
```
dram_init() # common/init/board_init.c:120
└─ k1_ddr_init() # board/spacemit/k1/ddr.c:45
├─ ddr_phy_init() # drivers/ddr/spacemit/phy.c:200
└─ ddr_training() # drivers/ddr/spacemit/training.c:80
```

## 关键数据结构
（结构体定义、字段含义、在代码中的传递路径）

## 关键逻辑
（条件分支、配置解析、硬件交互的详细分析）

## 开源/闭源边界 （如适用）
（哪些是开源代码、哪些调用了闭源 blob、接口是什么）

## 与其他模块的关系
（上游调用者、下游依赖、共享的全局变量/配置）

## 动态分发/隐性跳转
（函数指针 ops->init() 的实际绑定、宏展开路径、__weak 覆盖关系。
每个跳转记录：变量类型、赋值位置 file:line、实际目标函数。
无法静态解析的标注 [UNRESOLVED]，交由 Jarvis-Arch 在全局上下文中锚定。）

## 待分析
（看不懂的、需要进一步追的、需要硬件手册才能理解的）
引用格式规范
所有代码引用统一使用 文件路径:行号 格式（相对于代码库根目录）。
引用必须可验证 - Reviewer 可以直接 sed -n 'Np' file 核实。
如果代码库有多个分支/版本，在 PROJECT.md 中锁定分析的 commit hash。

SYMBOL_INDEX.md - 全局符号表
何时生成：Phase 1（侦察阶段）由 Jarvis-Arch 使用工具（ctags/cscope 或 LSP）生成。
内容：项目中关键的全局结构体定义位置、核心宏定义位置、全局变量分布。
价值：作为 Researcher 的常驻上下文，极大降低寻找定义时耗费的检索步骤和幻觉率。
维护：Jarvis-Arch 在整合每个模块时按需补充新发现的关键符号。
decisions.md - 分析决策库
用途：记录架构级别的"为什么"。（例如："为什么我们决定跳过 drivers/usb 目录？" 或 "为什么判定 xxx 函数为死代码？"）
价值：当后期回顾或 Guodong 进行 Sign-off 时，防止 Agent 回答"我不记得为什么当时这么决定了"，确保分析过程的不可辩驳性。
knowledge-map.md - 全局知识图谱
开发框架没有的新概念。代码分析是渐进式拼图 - 每个模块的分析结果会修正/补充对其他模块的理解。

知识图谱记录：
- 已确认结论：经过 Reviewer 验证的事实（附代码引用）
- 模块关系图：模块间的调用关系、数据流（mermaid）
- 开放问题：跨模块的未解之谜、需要后续模块分析才能回答的问题
- 术语表：项目特有的缩写、命名约定

Jarvis-Arch 在每个模块分析通过后更新知识图谱。
Researcher 在开始新模块分析前必须先读知识图谱，获取前序模块的上下文。

4.2 黑板状态机 (Blackboard State Machine)

在多智能体系统中，为了防止 Agent 互相聊天导致流程"脱轨"，框架不依赖 Agent 的自觉性，而是强制引入基于黑板状态机的流转协议。流程的推进唯一取决于文件状态的变更。

状态标签定义 (tasks-backlog.md)
每个模块分析任务必须带有以下生命周期标签：

⏳ [TODO] - 待分配
🏃 [Analyzing] - Jarvis-Arch 已将任务写入 task.md，Researcher 正在处理
🔍 [Reviewing] - Researcher 已输出分析文档，Reviewer 正在核查
✅ [Done] - Reviewer 签署通过，Jarvis-Arch 已更新知识图谱
⚠️ [Needs Re-evaluation] - 因依赖的底层模块结论发生改变，触发的退回重审
🔒 [Escalated] - 超过 3 轮未决，必须由 Guodong 介入

[Escalated] 恢复机制：老板介入后，通过更新 task.md（补充说明、缩小范围）或写 decisions.md（裁决争议点）来提供指导。Jarvis-Arch 收到老板指示后，将状态从 [Escalated] 改回 [Analyzing]（需 Researcher 重做）或 [TODO]（需重新派发），流程恢复。老板未明确指示前，[Escalated] 状态不得被任何 agent 修改。
强校验执行闭环 (The Enforcement Loop)
1. 主控轮询：Jarvis-Arch 检查 backlog。看到 [TODO]，改为 [Analyzing]，写好 task.md，唤醒 Researcher
2. 物理文件交接：Researcher 被唤醒后，只准读 task.md。分析完后，必须生成分析文档，并亲自去 backlog 里把状态改为 [Reviewing]
3. 事件触发：Jarvis-Arch 监测到状态变成 [Reviewing]，立刻拉起 Reviewer
4. 防扯皮机制：Reviewer 和 Researcher 的争论必须写在物理文件 review.md 里。超过 3 轮未决，状态被锁定为 [Escalated]，必须由 Guodong 介入
5. 断点续传：就算中途系统断电、进程崩溃，重新启动后 Jarvis-Arch 只要看一眼 tasks-backlog.md 上的标签，就能立刻知道接下来该拉起哪个 Agent 继续干活

5. Reviewer 审查方法

代码分析审查 ≠ 代码开发审查
开发框架的 Reviewer 检查"代码是否符合 spec"。
分析框架的 Reviewer 检查"分析是否符合源码"。方向相反。

5.1 审查清单

检查项方法严重程度
代码引用准确性抽查 ≥30% 的代码引用（文件:行号），验证函数名/签名是否匹配🔴 P1 - 引用错误直接打回
调用链完整性检查是否有遗漏的分支、条件编译(#ifdef)、错误处理路径🔴 P1 - 主路径遗漏必须补
数据流准确性追踪关键参数的传递路径，验证 Researcher 的描述🟡 P2 - 非核心路径可标注待分析
结论有据每条结论是否有对应的代码引用？有无无证据的推测？🔴 P1 - No Evidence, No Claim
隐性跳转处理函数指针、ops 结构体、宏替换是否正确解析？未解析的是否标注 [UNRESOLVED]？🔴 P1 - 静默跳过等于断头路
待分析标注看不懂的部分是否诚实标注？有无假装理解的嫌疑？🟡 P2
与知识图谱一致是否与前序模块的已确认结论矛盾？🟡 P2 - 矛盾需要明确记录

5.2 强制工具验证 (Mandatory Tool Use)

🔴 对抗幻觉引用 (Hallucinated Citations)
LLM 极易伪造看起来合理的 文件:行号 引用。如果 Reviewer 仅靠"阅读"去审查（纯 LLM 模式），它极容易被 Researcher 伪造的完美逻辑"催眠"并放行。

强制规则：Reviewer 必须通过执行 CLI 命令验证代码引用，禁止纯文本审查。

验证方式：
- sed -n '行号p' 文件路径 - 抓取物理文件中的实际代码行
- grep -n '函数名' 文件路径 - 验证函数是否存在于该文件
- 将 CLI 返回的实际代码与 Researcher 的描述做语义比对
- review.md 中必须贴出 CLI 命令及其输出（可审计的验证证据）

Reviewer SOUL.md 必须配置 exec 工具权限。没有工具权限的 Reviewer 等于没有 Reviewer。

5.3 对抗"分析催眠" (Analysis Hypnosis)

代码分析也有催眠效应 - Researcher 写了大段看似合理的分析，Reviewer 顺着读下来觉得都对。

强制抽离规则：
- Reviewer 不能只读分析文档，必须通过 CLI 命令打开源码逐项核对
- 每次 review 至少自己独立追一条调用链（从入口追 2-3 层），与 Researcher 的描述对比
- 如果 Reviewer 的独立追踪与 Researcher 的分析完全一致且没有任何新发现，应怀疑自己是否被催眠了 - 换一条不同的路径再追一次
- review.md 必须包含 Reviewer 自己的代码引用和 CLI 验证输出（证明确实打开了源码看）
- #ifdef 强制交叉验证：遇到 #ifdef / #if defined() 包裹的代码块时，Reviewer 必须先查 PROJECT.md 中的目标编译配置确认该宏的状态（活跃/非活跃），不可自行推断宏的真假。review.md 中须注明所依据的 CONFIG 状态

5.4 证据优先原则 (Evidence Priority)

当 Reviewer 使用 CLI 命令（sed/grep）产出物理证据证明 Researcher 的引用或结论有误时，Reviewer 的物理证据优先于 Researcher 的语义推论。

Researcher 不得与带有 CLI 验证输出的否定意见争辩 - 应根据证据修正自己的分析。

这条规则确保了 review 争议的裁决依据是可验证的物理事实，而非谁的论证更"合理"。

预处理条件约束：涉及 #ifdef 条件编译区域的代码引用，验证方必须基于 PROJECT.md 中记录的目标编译配置判断该代码路径是否活跃。对死代码路径（当前编译配置下不会编译的代码）的引用不构成有效证据。

5.5 review.md 格式

轮次计数器（必须）：LLM 无状态，无法自行记忆当前是第几轮 review。Reviewer 每次驳回必须在 review.md 标题中写明显式轮次 [REJECT, Round X/3]。Researcher 修正后回复时也必须带轮次号。这是 3 轮熔断机制能被正确执行的前提。

## Round 1 (2026-03-10 16:00) [REJECT, Round 1/3]
结论：❌ 打回

- ✅ dram_init() @ common/init/board_init.c:120
验证: $ sed -n '120p' common/init/board_init.c
输出: int dram_init(void) { ← 匹配
- ❌ k1_ddr_init() @ board/spacemit/k1/ddr.c:45
验证: $ sed -n '45p' board/spacemit/k1/ddr.c

### 独立追踪

### 问题清单

6. 代码访问策略

代码库不 clone 到项目目录
代码库可能很大（Linux kernel、U-Boot 等），不应 clone 到每个 agent 的 workspace。

访问方式（按优先级）：
1. 本地 clone：如果已 clone，在 PROJECT.md 中记录路径，agent 直接读文件
2. GitHub API：通过 api.github.com 或 raw.githubusercontent.com 按需读文件
3. web_fetch：对于其他代码托管平台

在 PROJECT.md 的 代码库信息 段记录：
- 仓库 URL
- 分析的 commit hash 或 tag（锁定版本）
- 本地 clone 路径（如有）
- 访问方式（local / github-api / web）
- 目标编译配置：defconfig 名称 + 关键 CONFIG 选项（决定 #ifdef 分支走向，所有 agent 必须基于同一配置分析）

路径准确性规则：所有 CLI 命令（sed/grep/cat）必须使用相对于代码库根目录的完整路径，禁止依赖当前工作目录（CWD）。大型项目（如 Linux kernel）存在大量同名文件，依赖 CWD 会导致读错文件、引用失效。

7. 知识图谱维护

渐进式拼图 vs 一次性交付
开发框架的每个 task 是独立的 - 模块 A 完成不影响模块 B 的实现。
代码分析不同 - 分析模块 B 时可能发现模块 A 的分析有误或不完整。

处理方式：
- Researcher 在分析新模块时发现前序模块的问题 → 在"待分析"中记录，不自行修改前序文档
- Jarvis-Arch 在整合知识图谱时决定：是回头修正前序文档，还是在知识图谱中标注修正
- 重大修正（推翻前序结论）→ 通知 Guodong

🔴 依赖传导机制 (Dependency Propagation)：
当某核心结论被修正时，依赖该结论的下游模块可能变成污染数据。

knowledge-map.md 中的每条"已确认结论"必须记录被哪些模块引用。一旦该结论被修正：
- Jarvis-Arch 自动将所有引用该结论的模块标记为 [Needs Re-evaluation]
- 在 tasks-backlog.md 中为这些模块创建重审任务（优先级高于新模块分析）
- 重审范围：仅限受影响的结论链，不需要全部重做
- 通知 Guodong 级联影响范围

🔒 防循环重审死锁 (Anti-Deadlock)：
C 代码中模块间常存在循环依赖（如：中断模块 A 依赖电源模块 B，B 的异常处理又回调 A）。级联刷新可能导致 A→B→A 的无限 Ping-Pong。
规则：当检测到同一模块被重审超过 2 次，强制将状态锁定为 [Escalated] 并中止流转，交由 Guodong 进行人工架构仲裁。可选方案：将两个循环依赖的模块合并为一个联合分析任务。

8. 复盘与进化

沿用开发框架的复盘机制，但复盘三问针对分析场景调整：

问题检查什么更新什么
1. 分析效率如何？Researcher 追调用链的策略是否高效？有没有走太多弯路？task.md 模板（入口点描述是否足够）
2. 审查发现了什么？引用错误率、遗漏率、催眠率Reviewer SOUL.md Learned Rules
3. 知识图谱需要修正吗？新模块分析是否推翻了旧结论knowledge-map.md
SOUL.md 与 MEMORY.md 权责分离
SOUL.md 是只读的：只有 Guodong 能修改。任何 Agent（包括 Jarvis-Arch）禁止修改任何人的 SOUL.md。让 LLM 修改自己的系统提示词是危险的，容易写坏格式或引入偏差。

MEMORY.md 是经验外挂：每次项目复盘产生的教训，由 Jarvis-Arch 写入对应 Agent 目录下的 MEMORY.md。

启动时必须读 MEMORY.md：每个 Agent 在开始任何任务前，必须先读取自己目录下的 MEMORY.md 中的历史教训。

经验 ≠ 事实：MEMORY.md 记录的是历史教训和模式总结，不是代码事实。当 MEMORY.md 中的经验与当前代码的实际观测矛盾时，以当前代码事实为准。发现矛盾的 agent 应报告给 Jarvis-Arch，由 Jarvis-Arch 统一更新 MEMORY.md（保持单一写原则）。

如果某条经验足够重要需要写入 SOUL.md，向 Guodong 提议，由他审批后修改。

8b. 大型项目扩展 (Scaling)

以下机制在中小型项目中不是必须的，但在分析 Linux Kernel、U-Boot 等巨型代码库时至关重要。

分级地图 (Tiered Knowledge Map)
当项目极大（如 Linux Kernel 成百上千个驱动）时，单一 knowledge-map.md 会膨胀到十几万字，导致 Token 爆炸和 Agent 抓错重点。

分级方案：
- global-map.md：极精简，只记录子系统间的接口、调用关系和共享数据结构
- .blackboard/subsystem-maps/：按业务域拆分（如 usb-map.md、memory-map.md），记录子系统内部细节

生成规则：
- Phase 1 决定：Jarvis-Arch 侦察代码库后，与人类工程师讨论决定是否需要分级、如何划分子系统边界。决定记录在 PROJECT.md 中
- 初始生成：Phase 1 由 Jarvis-Arch 生成 global-map.md 初始框架和 subsystem-maps/ 目录结构

所有权与更新：
- 只有 Jarvis-Arch 能修改 global-map.md 和 subsystem-maps/*.md
- Phase 2 每个模块完成后，Jarvis-Arch 增量更新对应的 subsystem map
- 接口级变化（子系统间新增调用关系、共享数据结构变更）同步到 global-map.md，子系统内部细节不传导

Researcher 读取规则：
- Researcher 只读 task.md 里显式指定的 map，不自行加载其他 map
- Jarvis-Arch 在每个 task.md 中明确列出该任务需要加载的 map 文件（global-map.md + 相关 subsystem map）
上下文剪枝 (Context Pruning)
分析大型代码库时，Researcher 频繁读取源码会快速耗尽 Token 预算。Jarvis-Arch 负责上下文管理：

- task.md 精准聚焦：每个 task.md 必须明确限定分析范围和入口点，避免 Researcher 发散
- knowledge-map.md 只保留核心：已确认结论只记系统级契约和核心接口，模块内部细节留在 analysis/*.md
- Researcher 上下文隔离：每个 Researcher session 只加载当前模块的 task.md + knowledge-map.md + SYMBOL_INDEX.md，不加载其他模块的 analysis/*.md（除非 task.md 显式指定）
- 大文件处理：Researcher 读源码时应按函数/代码块精确定位（sed -n 'start,endp'），禁止整文件读取

9. 部署与运转 (Deployment & Operations)

💡 最佳实践：前台路由模式 (Router Pattern)
如果你已有日常助手（如 Jarvis），不建议让它直接承担 Jarvis-Arch 的职责 - 日常助手应保持轻量。
最优雅的做法：让现有日常助手充当"前台调度员"，遇到代码分析任务时原封不动地路由转交给 Jarvis-Arch。
这样既保持了日常助手的轻量和懂你，又确保了框架执行的严谨性。

9.1 部署顺序："先有灵魂，再造躯体"

第一步：撰写并挂载 SOUL.md
将 §11 的模板保存至对应目录（如 ~/.openclaw/agents/researcher/SOUL.md）。

第二步：注册并配置 Agent（精确挂载工具权限）
- Researcher：只读文件系统权限（cat, ls, grep）+ 知识图谱读取权限
- Reviewer：必须挂载 CLI 工具执行权限（sed -n, grep），使其具备物理校验能力
- Jarvis-Arch：唤醒/调用其他 Agent 的编排权限 + 读写 .blackboard/ 目录的完整权限

第三步：日常运转（全自动流转）
1. 下达指令：Guodong 直接对 Jarvis-Arch 说"帮我逆向分析一下 K1 项目的 DDR 模块"
2. 任务生成：Jarvis-Arch 与 Guodong 对齐目标后，生成 backlog，唤醒 Researcher
3. 自我博弈：Researcher 分析完毕修改状态为 [Reviewing]，触发 Jarvis-Arch 唤醒 Reviewer 找茬（最高 3 轮）
4. 并发提效：若有多个无依赖子模块，可拉起多个 Researcher 并行分析
5. 验收交付：达成一致后，Jarvis-Arch 整合 knowledge-map.md，推送给 Guodong 验收

9.2 项目收尾与经验传承 (Project Wrap-up & Evolution)

Agent 与项目完全分离 = 铁打的团队，流水的项目
项目分析完成后，不要删除 Agent 实例。Jarvis-Arch、Researcher、Reviewer 是你的全职核心团队，不同的代码库只是他们工作台上的不同案卷。

保留 Agent（铁打的团队）：
SOUL.md 是通用的，长期驻留在系统的全局目录中随时待命。

切换项目上下文（流水的项目）：
下次有新代码库需要分析时，只需创建新项目文件夹（如 ~/projects/new-project/），初始化新的 PROJECT.md。直接对 Jarvis-Arch 说"接新活了"，原班人马走到新项目的黑板前开工。

💡 终极红利：经验累积 (Evolution)
将每个项目复盘得到的教训（例如特定厂家的宏混淆规律、Reviewer 容易漏看的陷阱），补充到对应 Agent SOUL.md 底部的 == 启动检查清单 == 中。
Agent 团队会随着分析项目的增多跨项目沉淀"肌肉记忆"，变得越来越刁钻和老练。

10. 设计决策汇总

D1: Source-Driven Analysis - 源码是唯一 source of truth
分析文档的每条结论都必须能追溯到源码。No Evidence, No Claim。
D2: 三角色 - Jarvis-Arch(协调) + Researcher(分析) + Reviewer(验证)
Jarvis-Arch 不亲自做深度代码分析（拆分和整合已经够忙），Researcher 专注读代码，Reviewer 专注验证准确性。
D3: 知识图谱 - 渐进式拼图
每个模块分析完成后更新全局知识图谱。新模块分析可能修正旧结论。知识图谱是跨模块的共享上下文。
D4: 两阶段 - 侦察在前，深度分析在后
Phase 1 画地图（PROJECT.md + 模块拆分），Phase 2 逐模块深挖。先搞清楚全貌再钻细节。
D5: 最多 3 轮 review（同开发框架）
3 轮未过升级给 Guodong。防止 Researcher 和 Reviewer 死循环。
D6: Agent 与项目分离（同开发框架）
SOUL.md 定义通用角色，PROJECT.md 定义项目信息。换项目不用重建 agent。
D7: 代码库不 clone 到 agent workspace
大型代码库（kernel、U-Boot）太大。用本地已有 clone 或 GitHub API 按需访问。
D8: 先串行，后并行（同开发框架）
先串行跑通完整流程。并行扩展方案：tasks/ + reviews/ 目录，多个 Researcher 实例分析无依赖的模块。
D9: Reviewer 必须独立追踪
每次 review 至少自己独立追一条调用链，不能只读分析文档打勾。对抗分析催眠。
D10: Reviewer 必须使用 CLI 工具验证（对抗幻觉引用）
LLM 极易伪造合理的代码引用。Reviewer 必须通过 sed/grep 等 CLI 命令物理验证代码行，禁止纯文本审查。review.md 中必须包含 CLI 命令及输出作为可审计证据。
D11: 隐性跳转必须显式记录（对抗断头路陷阱）
C 代码中的函数指针、ops 结构体、宏替换、__weak 覆盖等静态无法解析的跳转，Researcher 必须显式记录并标注 [UNRESOLVED]，交由 Jarvis-Arch 在全局上下文中锚定。禁止静默跳过。
D12: 知识图谱依赖传导（对抗陈旧级联）
已确认结论必须记录被哪些下游模块引用。结论被修正时，Jarvis-Arch 自动将依赖模块标记为 [Needs Re-evaluation]，触发局部重审。防止污染数据级联扩散。
D13: 防循环重审死锁（对抗模块循环依赖）
模块间循环依赖可能导致级联刷新无限 Ping-Pong。同一模块被重审超过 2 次时，强制锁定为 [Escalated]，交由 Guodong 仲裁或合并为联合分析任务。
D14: SOUL.md 只读，经验写入 MEMORY.md
SOUL.md 是角色本质定义，只有 Guodong 能修改。复盘产生的经验教训由 Jarvis-Arch 写入各 Agent 的 MEMORY.md。Agent 启动时必须先读 MEMORY.md。避免 LLM 修改自己的系统提示词。
D15: 大型项目知识图谱分级
当项目规模极大时，单一 knowledge-map.md 会 Token 爆炸。分级方案：精简 global-map.md + 按业务域拆分 subsystem-maps/。Phase 1 由 Jarvis-Arch 与人类工程师讨论决定。详见 §8b。

11. 流程协议分层 (Protocol Layering)

11.1 问题

K1 U-Boot SPL 实战暴露了一个核心问题：agent 不遵守设计者制定的流程协议。

具体表现：Jarvis-Arch 擅自将串行流程改为并行，违反了状态机设计。由此引发通知丢失、级联返工风险、老板被迫人工催促等连锁问题。

根因分析：
1. 设计文档不参与运行时 - design-architecture.html 写得很详细（流程、时序图、状态机），但 agent 从未读过它。它是给人看的，不是给 agent 看的。
2. SOUL.md 承担了过多职责 - 既管身份性格，又管流程规则，而且流程部分是 design-architecture.html 的压缩版，信息丢失不可避免。
3. 没有硬约束 - 所有流程规则都是提示词级别的软约束（"请遵守"），agent 可以自行判断"优化"而违反。

11.2 设计决策

D16: 职责分离 - SOUL.md vs PROTOCOL.md
SOUL.md 只管身份、性格、禁忌（"你是谁"）。
PROTOCOL.md 管流程、状态机、通信时序、guard 规则（"怎么做事"）。
SOUL.md 中原有的"通信协议"、"三阶段职责"等流程内容迁移到 PROTOCOL.md，SOUL.md 瘦身。
冲突时以 PROTOCOL.md 为准。
D17: PROTOCOL.md 跟团队走，不跟项目走
PROTOCOL.md 定义的是"这个团队怎么协作"，不会因分析的项目不同而变。因此它跟团队走，部署在 agent workspace 目录内。

部署方式：PROTOCOL.md 存放在 ~/.openclaw/agents/shared/PROTOCOL.md，三个 agent 的 workspace 内通过 symlink 指向同一份文件。
维护链路：design-architecture.html（设计源）→ PROTOCOL.md（agent 可读版）→ 自动注入到执行上下文。
更新时：改 ~/.openclaw/agents/shared/PROTOCOL.md 一个文件，三个 agent 同时生效。

与项目级文件（PROJECT.md、tasks-backlog.md 等）的区别：
- PROTOCOL.md = 团队协作规范（部署一次，跨项目复用）
- PROJECT.md / .blackboard/* = 项目信息（每次新项目由 Jarvis-Arch 创建）
D18: AGENTS.md 作为自动注入的启动指令
OpenClaw 会将 AGENTS.md 自动注入到每个 session（包括 sessions_spawn 创建的 session），这是唯一可以 100% 保证 agent 看到的投递渠道（除 SOUL.md 外）。

利用这一机制，为每个 agent 部署一份极简的 AGENTS.md，核心作用是强制 agent 启动时读取本目录下的 PROTOCOL.md。替换 OpenClaw 默认 AGENTS.md 模板中与 source-TRIO 无关的通用内容。

AGENTS.md 内容：
# AGENTS.md
在执行任何操作之前，必须先读取以下文件：
1. SOUL.md - 你的身份
2. PROTOCOL.md（本目录下）- 流程协议，权威文档，一切流程以此为准
未读 PROTOCOL.md 就开始工作 = 违规。
D19: guard.sh - 关键规则的代码级硬约束
在状态变更前，agent 必须调用 guard.sh 做合规检查。脚本返回非零退出码则禁止操作。
例：已有 [Analyzing] 状态的任务时，禁止将新任务改为 [Analyzing]（串行强制）。
guard.sh 跟 PROTOCOL.md 一起部署在 ~/.openclaw/agents/shared/，symlink 到各 agent workspace。PROTOCOL.md 中写明调用规则。
虽非 100% 防弹（agent 仍可跳过），但将规则从"理解文字"变为"运行命令看结果"，LLM 遵从率显著提高。

12. Agent 人格定义 (SOUL.md)

每个 agent 的完整 SOUL.md 在 agents/*/SOUL.md 文件中。agents/*/SOUL.md 是 SOUL 的唯一 source of truth，本章只记录设计意图和关键设计决策，不重复全文。

维护链路：修改 SOUL → 直接改 agents/*/SOUL.md，本章无需同步（除非设计意图变了）。

12.1 Researcher - "数据流探长"

设计意图：只信编译器实际执行的指令，不信注释。追踪数据流变化而非仅记录调用关系。

关键设计决策：
- 数据流高于调用流（D-R1）：不仅报告"A 调用了 B"，必须说明核心数据结构在链路中发生了什么变化
- 承认无知（D-R2）：禁用"可能"、"大概"等模糊词，遇到无法追踪的跳转用 [PENDING] / [UNRESOLVED] 标签
- 对抗注释欺骗（D-R3）：TODO/FIXME/陈旧注释视为不可信线索
- 证据优先（§5.4）：Reviewer 的 CLI 物理证据优先于 Researcher 的语义推论

完整定义：agents/researcher/SOUL.md

12.2 Reviewer - "逻辑审判官"

设计意图：有罪推定 - 默认所有分析报告充满幻觉，直到 CLI 物理验证证明无误。

关键设计决策：
- 零信任 + 强制工具验证（D10）：所有代码引用必须通过 sed/grep 物理验证，纯文本阅读不算验证
- 对抗分析催眠（D9）：逆向验证 + 影子追踪，禁止顺着报告线性阅读
- 语境毒性检查（D-V1）：检查引用代码是否在死代码区（#ifndef 屏蔽、永假 if 分支）
- 无情打回（D-V2）：宁可升级老板也不放行有逻辑断层的报告

完整定义：agents/reviewer/SOUL.md

12.3 Jarvis-Arch - "知识压缩器与状态机"

设计意图：剥离噪音、驱动流程。基于文件状态机调度，不依赖记忆。

关键设计决策：
- 状态机驱动（D-A1）：所有调度以 tasks-backlog.md 状态标签为唯一依据
- Token 预算管控（D-A2）：knowledge-map.md 只存系统级契约和核心接口，非全局信息不得进入
- 冲突熔断（D-A3）：新旧结论冲突时不覆盖、不丢弃，生成 Conflict Report 交由老板裁定
- 清晰任务边界（D-A4）：通过关注点分离控制 Researcher 的上下文窗口，防止幻觉

完整定义：agents/jarvis-arch/SOUL.md
